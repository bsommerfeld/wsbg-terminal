## ⚠️ MANDATORY TO SMOKE TEST VIA `run.sh` (vor jeder Bewertung)

Der automatisierte Live-IT beweist nur die **Mechanik**, NICHT den redaktionellen
Wert. Bevor irgendjemand „klappt / besser als master" beurteilt, MUSS die echte
Wire im laufenden Terminal gesehen werden:

```bash
./.script/run.sh
```

**Worauf zu achten ist (das ist die eigentliche offene Frage):**
1. **Theme vs. Units auf der Wire** — pro Reddit-Thread erscheint jetzt EINE
   Theme-Zeile (Thread-Narrative, gekeyt auf cluster-id) ZUSÄTZLICH zu den
   per-Subjekt Unit-Zeilen. Liest sich das als **Mehrwert** (der Thread-Kontext,
   den die Units nicht haben) oder als **Doppelung/Rauschen**?
2. **Volumen** — die Theme-Zeilen ~verdoppeln die Zeilenzahl pro Thread. Ist das
   im 1:1-Spiegel-Sinn ok oder zu viel?
3. **Ko-Okkurrenz-Fall** — ein Thread, der zwei Dinge nebeneinanderstellt
   (z. B. Waffenstillstand/Frieden + ein Aktien-Chart darunter): trägt die
   Theme-Zeile die Narrative („… dazu Chart X −8% gepostet"), die die Units
   strukturell nicht haben? **Das ist der Daseinszweck des Theme-Produzenten.**
4. **Kausal-Guard** — schreibt der Theme „im selben Thread"/„dazu" und NICHT
   „wegen"/„weil"? (Erfundene Kausalität wäre ein Regressionsbefund.)

Optional zusätzlich der Lab-Harness mit konstruiertem Ko-Okkurrenz-Thread:
```bash
./.script/run-test.sh   # Link eines Friedens-Threads mit Chart-Bild einfügen, „Los"
```

## ✅ Was PROVEN ist (automatisiert)
- Voller Build über 16 Module grün; **177 agent-Unit-Tests grün**.
- **`PipelineStagesIT` 3/3 live (gemma4:e4b, ~100 s):** ein dirty Cluster →
  `1 theme + 8 unit headlines`. Theme unter cluster-id, Units unter unit-id,
  beide koexistieren (kein Dedup). Theme-Assertion grün.
- Mechanik aller drei Stufen: cluster==thread (kein Reclustering), Theme-Compose
  + publish unter cluster-id (erbt ReportBuilder-Coverage), eine Theme-Zeile pro
  dirty Cluster pro Tick (kein Cap).

## ❓ Was UNPROVEN ist (deshalb der Mandatory-Smoke)
- **Redaktioneller Wert des Theme** im eigentlichen Fall (Ko-Okkurrenz) — der
  Auto-IT lief auf einem Aufzähl-Thread („nenn deine eine Aktie"), wo der Theme
  die Namen nur **gelistet** und die Units **gedoppelt** hat. Worst case für den
  Wertnachweis; der Zielfall wurde nie ausgelöst.
- **Kausal-Guard** — mangels Juxtaposition nie getriggert.
- **Coverage/UPDATE über mehrere Ticks** (re-dirty → Follow-up nur auf frische
  Kommentare) — Mechanismus ist der bestehende, getestete ReportBuilder-Pfad,
  aber nicht für den Theme über zwei Ticks assertiert.
- **„Besser als master"** — NICHT gemessen. Kein Side-by-Side. Objektiv ist der
  Branch nur *einfacher* (Embedding-Clustering + Rebalancer + Centroid raus);
  ob der OUTPUT besser ist, ist offen. Für eine harte Aussage: dieselben Threads
  durch master und Branch, Headlines nebeneinander.

## Offener Befund / Risiko
- **Theme↔Unit-Redundanz auf Listen-Threads** (live beobachtet): auf einem

  Annahme verfälschen). Ob das in der Praxis stört, entscheidet der `run.sh`-Blick.
