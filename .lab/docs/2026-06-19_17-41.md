# GEWÜNSCHTES VERHALTEN (Spec) + Branch-Audit gegen master

Auftrag: gesamte Codebase gegen master prüfen, Logik-Verschiebungen erfassen, das
gewünschte Verhalten dokumentieren. User-Direktive zur Architektur:
**Cluster und Einheiten gleich behandeln. Subjekte = Cross-Cluster-Einheiten.
Cluster = 1:1 zum Thread. Subjekt-Logik bleibt, Cluster wird daran angepasst.**

---

## 1. Gewünschtes Verhalten (die Referenz)

### Ingestion
- Reddit-Fetch über die `WebFetcher`-Chain (OAuth → browser → direct → RSS),
  self-healing, per-Instance-UA. 100% lokal.
- **Cluster = 1:1 zum Thread.** `ClusterEngine.assign`: cluster id == thread id,
  kein cross-thread Join, kein Merge/Prune (ClusterRebalancer entfernt). Ein
  Cluster ist ein treuer Wrapper EINES Threads. ✅ (bereits umgesetzt)
- Cross-Thread-Aggregation passiert **nur** auf Subjekt-Ebene: dasselbe Instrument
  aus N Threads → eine `SubjectUnit` (`SubjectRegistry.mergeIdentities`).

### Zwei Producer, ein Dirty-Signal (`runUnitTick`)
Ein dirty Cluster treibt ZWEI unabhängige, NICHT gegeneinander deduplizierte
Headline-Producer:
1. **Cluster-Theme** (das Thread-Narrativ, die Juxtaposition des Raums) — eine
   Headline pro dirty Cluster, **keyed by cluster id**. Prompt `headline-theme`.
2. **Subject-Unit** (das cross-cluster Subjekt) — eine Headline pro dirty Unit,
   **keyed by unit id**. Prompt `headline-compose-unit`, mit NEW/UPDATE.

### Wiederholungs-Schutz — MUSS für BEIDE robust + konsistent sein
Kern der Direktive. Beide Producer dürfen bereits verarbeitetes Material nicht
erneut als Headline aufrollen. **Referenz = Subjekt-Logik (zeit-basiert):**
- Evidenz, die *am/vor* der letzten Headline der Entität kam, ist „bereits
  berücksichtigt"; nur Material *danach* ist frisch. Markieren (Kontext bleibt),
  Follow-up nur aus dem Frischen. Robust, unabhängig von Modell-Zitaten.
- **Subject-Unit: ✅ erfüllt** (`unitBrief`, `[✓ COVERED]`-Tag gegen
  `UnitHeadline.atEpoch`).
- **Cluster-Theme: ❌ NOCH source-id-basiert** (`ReportBuilder` splittet
  fresh/covered anhand vom Modell zitierter `sourceThreadIds`/`sourceCommentIds`)
  → hängt daran, dass ein 4B-Modell zuverlässig zitiert (tut es oft nicht) →
  schwächerer Schutz, einzige harte Bremse ist der Identical-Text-Guard. **Muss
  an die zeit-basierte Subjekt-Logik angepasst werden** (Grenze = `createdAt` der
  letzten Theme-Headline des Clusters; ReportBuilder hat die prior headlines schon).

### Publish-Gates (für beide gleich, schon konsistent ✅)
`HeadlineWriter.publish` (Theme) bzw. `publishUnit` (Unit): HTML-Strip,
Identical-Text-Guard, IMPORTANT→NORMAL-Drossel, Source-ID-Hygiene, sane-price-cap,
Ticker/Snapshot resolver-validiert (nie vom Modell), `AgentStreamEndEvent`-Broadcast.

### Highlight-Rubrik
Rot = „Wo kann ich schnell Geld machen?" (konkrete, handelbare Gelegenheit).
NORMAL ist Default; Theme-Zeilen sind fast immer NORMAL. (Spätere Verschärfung
„IMPORTANT = ≥15% Move" ist im Control-Room dokumentiert, noch nicht aktiv.)

### Persistenz
Permanentes Headline-Archiv (JSONL) + 24h-Wire-Seeding; Kurz-TTL-Snapshot von
Reddit-Daten + Vision + Headlines + Cluster + **SubjectUnits** (verbatim).

## 2. Audit-Befunde (Branch vs. master)

Der Branch ist eine fast vollständige Neufassung/Erweiterung (46 Commits, 188
Dateien; master ist die alte Tool-Loop-Ära mit anderen Prompts). Module-Shifts:
neue `source-api`/`embedding`/`aggregator`/`finanznachrichten`; agent komplett
umgebaut; terminal JCEF+Bridges; Reddit hinter `RedditSource`+WebFetcher.

### ✅ Korrekt / gewünscht
- Cluster 1:1 Thread; ClusterRebalancer raus; Cross-Thread nur via Subjekte.
- Zwei-Producer-Tick verdrahtet (`AgentCoordinator` → `runUnitTick`).
- Unit-Wiederholungs-Schutz zeit-basiert (Fix `5bf785d`).
- Publish-Gates für beide Producer konsistent.
- SubjectUnit-Snapshot (Fix `0ed3fd6`); WebFetcher-Unifikation (`1a665ca`).

### 🔴 Inkonsistenz / Bug (die Direktive)
- **Cluster-Theme-Wiederholungs-Schutz ist citation-basiert, nicht zeit-basiert**
  → inkonsistent mit Units, schwächer. ReportBuilder-Coverage feuert nur, wenn das
  Modell beim letzten Mal Quellen zitiert hat. Anzupassen an die zeit-basierte
  Subjekt-Logik. **Das ist der konkrete „Cluster anpassen"-Task.**

### ⚠️ Cleanup (kein Funktionsfehler)
- **Toter Pfad:** `runTick`(430)/`processCluster`/`composeOne`/`runClusterTraced`
  + Prompt `headline-compose-single.txt` — vom Zwei-Producer-`runUnitTick`
  abgelöst, von Prod ungenutzt. (`runClusterTraced` war der Lab-Trace; Lab nutzt
  jetzt `attributeCluster`+`composeUnit`.)
- **Doppelter Brief-Bau** pro Cluster/Tick: `attributeCluster` und
  `publishClusterTheme` bauen je einen `buildReportData` — einmal bauen, an beide
  geben (Micro-Opt).
- `ClusterEngine` hält den Centroid + `EmbeddingService` nur noch für die
  Snapshot-Shape (routet nichts mehr) — Cleanup-Kandidat.
- Methodenname `runUnitTick` ist nach der Theme-Erweiterung leicht irreführend
  (treibt beide Producer) — ggf. `runTick` (nach Entfernen des alten) übernehmen.

### 📄 Bekannte Limits (kein Bug)
- SubjectAttributor TODO(B) semantisches Line-Matching; Resolver-Venue; Yahoo nur
  per Browser-Joker (Dev-IP geblockt); inertes Scaffolding (Aggregator/RssIndex/Fn).

### Logs
Einzige Terminal-Logs (`logs/terminal/latest.log`) sind vom 1. Juni (Boot+Shutdown,
0 Headlines, altes Snapshot-Format) — spiegeln den aktuellen Branch NICHT. Echte
Laufzeit-Logs brauchen einen frischen Run (= der Smoke).
